## Overview

Content blocks allow marketers to reuse the same content across multiple campaigns. Instead of recreating standard elements for every message, you can create a block once—such as a header, footer, or a designed call-to-action button—and reference it everywhere.

This API enables you to programmatically fetch, create, and update these content blocks on your MoEngage dashboard. 

<Info>
  Content blocks support nested structures. You can include existing blocks within new ones to create complex, modular templates. [Read more about Content blocks in MoEngage](https://help.moengage.com/hc/en-us/articles/4414768369172-Content-blocks-).
</Info>

## Content Block API Endpoints

The Content Block API consists of the following operations:

- [Create Content Block](/api-reference/content-blocks/create)
- [Update Content Block](/api-reference/content-blocks/update)
- [Get Specific Content Blocks](/api-reference/content-blocks/get-by-ids)
- [Search Content Blocks](/api-reference/content-blocks/search)

## Supported Content Types

| Content Type | Description |
| :--- | :--- |
| **HTML** | Rich text and code-based content, ideal for Email campaigns. |
| **TEXT** | Plain text content, ideal for SMS or Push notifications. |

## Data Center Configuration

The API endpoint varies based on your MoEngage Data Center (DC). Replace `{dc}` in the base URL with your specific DC number (e.g., `api-01`, `api-02`, etc.).

**Base URL:** `https://api-0{dc}.moengage.com/v1/external/campaigns`

## Notes

<Note>
**Authentication:**
All requests must include the `MOE-APPKEY` in the header. You can find your Workspace ID (App Key) in the MoEngage Dashboard under **Settings** -> **Account** -> **APIs**.
</Note>

<Note>
**Teams & Permissions:**
If your account has **Teams** enabled, passing `team_ids` is mandatory during creation to ensure the block is accessible to the correct users.
</Note>

## FAQs

### Managing Content Blocks
<AccordionGroup>
  <Accordion title="How do I use nested content blocks via API?">
    When creating or updating a block, use the `content_block_used` array. Provide the names of the existing content blocks you are nesting within the new block. If not using nesting, pass an empty array: `[]`.
  </Accordion>

  <Accordion title="What is the difference between content and raw_content?">
    `raw_content` refers to the original, non-personalized content provided during creation. When fetching blocks by ID, you can set `is_raw_content_required` to `true` to retrieve this original version for editing purposes.
  </Accordion>

  <Accordion title="Can I archive a content block via API?">
    The API supports `ACTIVE` and `DRAFT` statuses during creation/update. While the search API can filter for `ARCHIVED` blocks, status transitions to 'Archived' should typically be managed via the MoEngage Dashboard.
  </Accordion>

  <Accordion title="Can I search for blocks created by a specific user?">
    Yes. The Search API allows you to filter results using the `updated_by` field, which accepts an array of user email addresses.
  </Accordion>
</AccordionGroup>

### Personalization & Images
<AccordionGroup>
  <Accordion title="Does this API support personalization syntax?">
    Yes. You can include Jinja personalization (user attributes, event attributes, etc.) within the `raw_content` string. The syntax is identical to what is supported in the MoEngage dashboard.
  </Accordion>

  <Accordion title="How do I track which images are used in a block?">
    The `images_used` field accepts an array of URIs. This allows you to programmatically associate specific image assets with a content block for better asset management.
  </Accordion>
</AccordionGroup>
