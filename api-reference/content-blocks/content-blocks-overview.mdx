## Overview

Content blocks allow marketers to reuse the same content across multiple campaigns. For example, a block might be a repeatedly used header, footer, or designed call-to-action button.

This API enables you to programmatically fetch, create, and update content blocks on your MoEngage dashboard, ensuring brand consistency and reducing manual effort across teams.

<Info>
  If this API is not enabled for your account, contact your MoEngage Customer Success Manager (CSM) or the Support team to request enablement.
</Info>

## Content Blocks API Endpoints

The Content Blocks API consists of the following operations:

- Create Content Block
- Update Content Block
- Get Specific Content Blocks
- Search Content Blocks

Key Concepts

<Note>
  Nested Content Blocks: MoEngage supports nesting content blocks within each other. When creating or updating a block that contains other blocks, ensure you pass the names of the nested blocks in the content_block_used array to maintain proper dependency mapping.
</Note>

<Note>
  Team-Based Access: If your account uses Teams, passing team_ids is mandatory. This ensures the content block is available to the correct users within your organization. Refer to the Teams documentation for more details.
</Note>

FAQs

Management & Configuration

<AccordionGroup>
  <Accordion title="What is the difference between content and raw_content?">
    raw_content represents the original, non-personalized content provided during creation. The content field returned in search/get responses typically represents the processed version. When fetching specific blocks, you can use the is_raw_content_required flag to specifically retrieve the source content for further editing.
  </Accordion>
  <Accordion title="How do I uniquely identify a content block?">
    You can use the label field as a unique identifier for your content blocks. While the id is a system-generated MongoDB-style string, the label is often more readable for programmatic logic.
  </Accordion>
  <Accordion title="Can I use images in my content blocks?">
    Yes. When creating a block, you can pass an array of image URIs in the images_used field. This helps in tracking asset dependencies within the MoEngage ecosystem.
  </Accordion>
</AccordionGroup>

Search & Retrieval

<AccordionGroup>
  <Accordion title="Can I search for blocks updated by a specific user?">
    Yes. The Search API allows you to filter by updated_by using an array of email addresses. This is useful for auditing changes made by specific team members.
  </Accordion>
  <Accordion title="Is it possible to fetch multiple specific blocks in one call?">
    Yes. Use the Get Specific Content Blocks endpoint and pass a list of IDs in the ids array. This is more efficient than calling the API individually for every block.
  </Accordion>
  <Accordion title="How do I filter search results by status?">
    You can provide an array of statuses (e.g., ["ACTIVE", "ARCHIVED"]) in the search filters to narrow down your results to only relevant blocks.
  </Accordion>
</AccordionGroup>

Personalization

<AccordionGroup>
  <Accordion title="Can I include Jinja personalization in content blocks?">
    Yes. You can include Jinja syntax for user attributes, event attributes, and more within the raw_content. When the block is eventually rendered within a campaign, MoEngage will process the Jinja logic.
  </Accordion>
</AccordionGroup>
