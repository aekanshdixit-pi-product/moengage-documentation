# Developer Guide

The MoEngage SDK is a lightweight, modular client library that runs inside your application. It handles event tracking, user identification, push token registration, in-app message rendering, and content card delivery — syncing data to MoEngage servers via batched HTTPS requests with offline queuing.

```text
┌─────────────────────────────────────────────┐
│              Your Application               │
│  ┌───────────────────────────────────────┐  │
│  │          MoEngage SDK                 │  │
│  │  ┌──────────┐  ┌──────────────────┐   │  │
│  │  │ Analytics │  │  Push / In-App / │   │  │
│  │  │ & Events  │  │  Cards Engine    │   │  │
│  │  └─────┬─────┘  └────────┬────────┘   │  │
│  │        └────────┬─────────┘            │  │
│  │          ┌──────▼──────┐               │  │
│  │          │ Sync Engine │               │  │
│  │          │ (Batched +  │               │  │
│  │          │  Offline Q) │               │  │
│  │          └──────┬──────┘               │  │
│  └─────────────────┼─────────────────────┘  │
└────────────────────┼────────────────────────┘
                     │ HTTPS
              ┌──────▼──────┐
              │  MoEngage   │
              │   Backend   │
              └─────────────┘
```

## What the SDK Does

Once integrated, the MoEngage SDK gives your application these capabilities:

- **Analytics & user tracking** — track custom events, set user attributes, and manage user identity. Sessions, device info, app version, timezone, and locale are collected automatically.
- **Push notifications** — automatic push token registration and delivery handling across FCM, HMS, APNs, and Web Push. Rich templates, push amplification, and Live Activities where supported.
- **In-app messages & on-site messaging** — display targeted messages inside your app or website triggered by user behavior, events, or campaigns configured in the MoEngage dashboard.
- **Cards** — persistent content cards (notification center, inbox-style) that users can browse at their own pace. Supports built-in UI and self-handled rendering.
- **Offline resilience** — events and data are queued locally when the device is offline and synced automatically when connectivity is restored.
- **GDPR & compliance** — built-in data opt-out APIs for GDPR, CCPA, and other privacy regulations.

The SDK is modular on all platforms. You add only the features you need — there is no single monolithic dependency.

## Quick Start

Pick your platform and follow the integration guide. Estimated time to first event: **15–30 minutes** for native SDKs, **30–60 minutes** for cross-platform (due to additional native configuration).

<Steps>
  <Step title="Get your Workspace ID">
    In the MoEngage Dashboard, go to **Settings → Account → APIs → Workspace ID**. You'll need this to initialize the SDK.
  </Step>
  <Step title="Set your Data Center">
    Your data center determines the API endpoint the SDK communicates with. Find your data center from the dashboard URL host (e.g., `dashboard-01.moengage.com` → `dc_1`). This must be configured during SDK initialization.
  </Step>
  <Step title="Install the SDK">
    Follow the platform-specific installation link from the table below.
  </Step>
  <Step title="Initialize the SDK">
    Call the initialization method with your Workspace ID and data center. Use the **TEST** environment during development and **LIVE** for production.
  </Step>
  <Step title="Track your first event">
    Call the event tracking API to send a custom event. Verify it appears in the MoEngage Dashboard under **Analytics → Event Analysis**.
  </Step>
</Steps>

### Prerequisites by platform

Before starting integration, ensure you have:

- **Android:** Firebase project with `google-services.json` configured (required for FCM push). Gradle 7.0+, AGP 7.0+, Java 8+.
- **iOS:** Apple Developer account with push notification entitlement enabled. APNs Authentication Key (`.p8` file) uploaded to MoEngage Dashboard.
- **Web:** HTTPS-enabled domain (required for web push via Service Workers).
- **Cross-Platform (React Native / Expo / Flutter / Cordova / Capacitor):** All of the above native prerequisites, plus the framework-specific toolchain.

## SDK Performance & Data Sync

The MoEngage SDK is designed to have minimal impact on your app's performance, startup time, and battery life.

**Modular architecture:** You only include the modules you need. The core analytics module is lightweight; push, in-app, cards, and geofence are separate dependencies that add footprint only when used.

**Batched network calls:** The SDK batches events and syncs them periodically rather than making a network call per event. When the device is offline, data is queued locally and flushed automatically when connectivity resumes. No data is lost.

**Default data collection:** The SDK automatically captures session start/end, app version, device model, OS version, timezone, locale, and push tokens (when push is configured). All other data — custom events, user attributes — requires explicit API calls from your code.

For detailed size and performance measurements, see: [Android SDK size](/developer-guide/android-sdk/performance/sdk-size-impact) · [Android performance](/developer-guide/android-sdk/performance/sdk-performance) · [iOS framework size](/developer-guide/ios-sdk/framework-size-impact/framework-size-impact)

## SDK vs. API: When to Use Which

MoEngage provides both client-side SDKs and [server-side APIs](/api/introduction). Most integrations use both — the SDK on the client and the API for server-side operations. Both share the same Workspace ID.

| Use Case                                            | SDK           | API        |
| --------------------------------------------------- | :-----------: | :--------: |
| Track events from user's device                     | ✅             | —          |
| Track server-side events (e.g., payment processed)  | —             | ✅          |
| Register push tokens                                | ✅ (automatic) | ✅ (manual) |
| Show in-app messages                                | ✅             | —          |
| Send transactional alerts (OTP, order confirmation) | —             | ✅          |
| Manage user attributes from your backend            | —             | ✅          |
| Bulk import users/events                            | —             | ✅          |
| Create/manage campaigns programmatically            | —             | ✅          |

## FAQ

<AccordionGroup>
  <Accordion title="What data does the SDK collect automatically?">
    Session start/end, app version, device model, OS version, timezone, and locale. Push tokens are registered automatically when push is configured. All other tracking (custom events, user attributes) requires explicit API calls from your code. The SDK does not collect location data unless you explicitly integrate the geofence module.
  </Accordion>
  <Accordion title="Does the SDK work offline?">
    Yes. Events and data are queued locally when the device has no network connectivity. The SDK flushes the queue automatically when connectivity resumes. No data is lost.
  </Accordion>
  <Accordion title="What is the difference between TEST and LIVE environments?">
    TEST and LIVE are separate environments within the same MoEngage workspace. Use TEST during development to avoid polluting production analytics and segments. Switch to LIVE for production builds. The environment is set during SDK initialization — no code changes are needed beyond the init call.
  </Accordion>
  <Accordion title="Do I need both the SDK and the API?">
    For most integrations, yes. The SDK runs on the client (user's device) and handles event tracking, push, in-app, and cards. The API runs on your server and handles transactional messages, backend event tracking, bulk data imports, and campaign management. Both use the same Workspace ID.
  </Accordion>
  <Accordion title="How do cross-platform SDKs relate to the native SDKs?">
    React Native, Expo, Flutter, Cordova, and Capacitor SDKs are wrappers around the native Android and iOS SDKs. They provide a unified API in your framework's language (JavaScript, Dart, TypeScript) but delegate to the native SDK for actual functionality. This means you still need to complete native setup steps for push notifications, in-app rendering, and other platform-specific features.
  </Accordion>
  <Accordion title="Where do I find my Workspace ID and Data Center?">
    In the MoEngage Dashboard: **Settings → Account → APIs → Workspace ID**. Your data center is determined by your dashboard URL — for example, `dashboard-01.moengage.com` maps to `dc_1`. Both are required during SDK initialization.
  </Accordion>
  <Accordion title="How do I subscribe to SDK release notifications?">
    Follow the MoEngage SDK repositories on GitHub and enable release notifications. See [Subscribe to MoEngage SDK Releases](/developer-guide/release-notes/subscribe-to-moengage-sdk-releases/subscribe-to-mo-engage-sdk-releases) for setup instructions.
  </Accordion>
</AccordionGroup>
