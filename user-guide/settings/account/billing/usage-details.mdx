# Overview

The **Usage** page helps you understand and track your monthly usage across various billing metrics, such as Custom User Actions (Events), System-Generated Events, Monthly Active Users (MAUs), and more, across all workspaces. Usage data is displayed in a list format, grouped by metric category. This enables you to manage your usage patterns and build transparency in billing.

# Available Metrics

The following table lists the metrics available on the Usage page:

| Metric | Description |
| --- | --- |
| **Custom User Actions (Events)** | The total number of custom events tracked (for example, "Added to Cart") via your client-side SDK, server-to-server(S2S) or any other integration. This count also includes Billable Standard Events, which MoEngage tracks at your request (for example, "Notification Dropped"). For more information on events, refer [here](/user-guide/data/event-data/a-complete-guide-to-event-tracking). |
| **System-Generated Events** | The total number of events automatically generated by MoEngage to monitor activities/behaviours (for example, "App/Site Opened"). For more information, refer [here](/user-guide/data/event-data/derived-events-and-attributes). |
| **Total Tracked Events** | The combined total of all Custom User Actions and System-Generated Events. |
| **Mobile Active Users** | The number of unique users who interacted with your mobile app in a specified month. |
| **Web Active Users** | The number of unique users who interacted with your website in a specified month. |
| **Total Monthly Active Users (MAUs)** | The sum of Mobile & Web Active Users represents the total number of users who interacted with your integrated web & mobile apps. |
| **TV Active Users** | Total events (user actions or system-generated events) that are tracked across all channels. |
| **Emails Sent** | Number of email messages sent to your users, across all campaigns and flows. |
| **SMS Sent** | Number of SMS messages sent to your users, across all campaigns and flows. |
| **WhatsApp Messages Sent** | Number of WhatsApp messages sent to your users, across all campaigns and flows. |
| **Cards Sent** | Number of cards sent to your users, across all campaigns and flows. |
| **RCS Sent** | Number of RCS messages sent to your user, across all campaigns and flows. |
| **Landing Page Views** | Total number of non-unique impressions or views on all landing pages. |
| **All Profiles** | The total number of unique user profiles (both identified and anonymous) present in your workspace. |
| **Identified Profiles** | The total number of identified user profiles (users with any identifiers such as email, mobile number or unique ID) present in your workspace. |
| **Open Analytics Usage** | The total volume of monthly data scanned to execute your open analytics queries. |
| **Total Inform Messages** | The combined total of all messages sent across all channels (Push, Email, SMS, and WhatsApp) via Inform. |
| **Inform Push Messages** | The number of push notifications sent to your users via Inform. |
| **Inform Email Messages** | The number of email messages sent to your users via Inform. |
| **Inform SMS Messages** | The number of SMS messages sent to your users via Inform. |
| **Inform WhatsApp Messages** | The number of WhatsApp messages sent to your users via Inform. |

To access the _Usage_ page, perform the following step:

1. On the left navigation menu in the MoEngage dashboard, click **Settings** \> **Account** \> **Billing**.
2. On the **Billing** page, select the **Usage** tab.

## Analyze Your Usage Trends

The **Usage Trends** section allows you to visualize data patterns for different metrics, organized under categories such as Data, Engage, and others. This section explains how to use the filters and controls to analyze your data.

<Frame>
  ![Analyze Your Usage Trends 1](/images/analyze-your-usage-trends-1.gif)
</Frame>

### Step 1: Filter Your Data

Select the data you want to analyze using the filters at the top of the page:

- **Select workspaces/account:** Use the filter drop-down to choose the scope of data to view:
  - **Workspaces:** Choose one or more workspaces to include in the analysis. By default, your current workspace is selected. You will only see workspaces for which the following criteria are met:
    - They are mapped to the same Billing account as the current workspace.
    - You have the necessary permissions in these workspaces (refer to the _Permissions to access_ section above).
  - **Account:** View all metrics aggregated as the sum across all workspaces in your account. This option requires the _Manage account details_ permission.
- **Select date range:** Define the time period. You can choose _This month_, _Last month_, or a _Custom range_ (up to the last 12 months). The selected date range is also applicable for the chart/table view.
  <Info>
    - By default, the page shows data for the current calendar month.
    - Usage data is available from October 2025 onwards.
    - All metrics are tracked based on your workspace's configured timezone.
  </Info>

### Step 2: View and Interpret the Trends

After you've filtered your data, use the controls to customize your chart/table view. You can use these controls to explore any metric:

- **Metric Selection:** Select a relevant category, such as **Data**, **Engage**, **Events**, or **MTUs**.
- **Granularity:** View data aggregated **Monthly** or **Quarterly**.
- **Chart and Table View:** Switch between a line chart for quick visual analysis and a table for exact numbers.

For example, to see the trend of all custom events you've tracked over the last 3 months, perform the following steps:

1. In the **Data** section, in the category drop-down list, select the metric you want to view.\
   **Note**: The **Events** category is selected by default.
2. Set the granularity to **Monthly** or **Quarterly** (as needed).\
   **Note**: The **Monthly** granularity is selected by default.
3. Select the chart or table icons to switch between a visual graph and a detailed data table.\
   **Note**: The **Chart** view is selected by default.

The display shows the total count of different metrics in your selected timeframe:

## Create and Manage Alerts

Unexpected overages can be costly and difficult to anticipate. You can monitor your MoEngage services by setting up custom alerts. This allows you to stay proactively aware of your usage trends and receive notifications when they exceed a defined limit, helping you avoid unexpected overages. To create an alert, perform the following steps: This gives you time to adjust campaigns, throttle activity, or plan for an upgrade before charges are incurred.

1. Click **\+ Manage Alert** in the upper-right corner.  Click **\+ Manage Alert** in the upper-right corner.
   <Frame>
     ![Manage Alerts](/images/manage-alerts.png)
   </Frame>
   \
   The **Alert management** page appears.
2. Click **\+ Create alerts** in the upper-right corner. <img src="/images/moengage_bf5b0a.png" />\
   The **Create alert** page appears. <img src="/images/moengage_fd81e5.png" />
3. On the Create Alert page, enter the General details:

- **Alert name**: Enter a unique name for your alert.
- **Create Alerts On**: Campaign Stats is selected by default. To monitor usage and billing metrics, select Billing.
- **Send Alert On**: Select your primary notification channel (for example, Email).
- (Optional) Click **\+ Add alert destination** to configure Slack, MS Teams, or Webhooks as an alert destination. For more information, refer to [Alert Destinations](/user-guide/settings/reports-and-alerts/alert-management/getting-started/create-an-alert#alert-destinations).

4. Define the Alert condition:

- In the **Send alerts when** list, select the metric you want to monitor (for example, Total Tracked Events, Total Monthly Active Users (MAUs), Web Active Users). For a full list of available metrics, refer to the [Available Metrics](#available-metrics).
- Set the threshold logic (for example, Is greater than) and enter the numerical limit in the input box.

5. In the **Members** section, select the team members who should receive these notifications. Note: You can add any registered MoEngage UI user to the alert.
6. Click **Create**.\
   For more information, refer to [Alert Management](/user-guide/settings/reports-and-alerts/alert-management/getting-started/overview-alert-management).

### How Alerts Work

MoEngage monitors your usage daily. You receive a notification as soon as the daily scan identifies that your usage exceeds a defined threshold.

- Evaluation Time: MoEngage evaluates alerts daily.
- Data Scanned: MoEngage scans usage data accumulated up to the evaluation time and compares it against your set limits.

<Info>
  - You can edit and delete the alerts. For more information, refer [here](/user-guide/settings/reports-and-alerts/alert-management/manage-alerts/managing-alerts).
  - MoEngage offers role-based permissions to create alerts. For more information, refer to [Access Roles](/user-guide/settings/account/team-management/access-roles).
</Info>

# FAQs

<Accordion title="How does MoEngage calculate Monthly Tracked Users?">
  MoEngage calculates Monthly Tracked Users (MTUs) using the App/Site Opened event by summing the mobile MTUs and web MTUs for the selected month.

  - Mobile MTUs are the number of users who open the app on an Android or iOS device at least once a month.
  - Web MTUs are the number of users who open the website at least once a month.

  **Note**: A user's activity is counted twice if they use both a mobile and a web device. The number of activities can be 1 or 1,000.
</Accordion>

<Accordion title="What is Event Fair Usage Policy (FUP)?">
  MoEngage uses an Event FUP model. Event FUP is the average number of tracked events allowed per user, as defined in your contract (typically between 50 and 500). If your account exceeds this average limit, your Billable MTUs are recalculated.

  **Formula for calculating Billable MTUs: Total Monthly Active Users or Events/FUP (where Events could be Custom User Actions or Total Tracked Events).**

  For example, if your contract grants you an Event FUP of 100 and your account records 100 million events in a month, your billable MTUs will be recalculated as 1 million.
</Accordion>

<Accordion title="What is the MoEngage billing cycle?">
  A billing cycle is the interval of time between two consecutive billing statements. This represents the timeframe during which MoEngage bills you for your usage of MoEngage services regularly. The length of a billing cycle can vary depending on your subscription and contract. Contact your admin or dedicated customer success manager (CSM) to know your billing cycle.
</Accordion>

<Accordion title="Why is there a difference in the data in Billing and Key Metrics?">
  The Key Metrics Monthly Active Users (MAU) shows the MAU count based on the last 30 days. In contrast, Billing MTUs are calculated based on the actual number of days in the month, varying between 28, 29, 30, or 31, depending on the calendar month or leap year.
</Accordion>

<Accordion title="Why is there a difference in the data in the billing and segmentation MAU count?">
  The MTU data provided by MoEngage for the billing may not match the segmentation data because:

  - MoEngage calculates the billing for every account on the first day of every month. When a user merge occurs after the billing calculation, re-running the query on segmentation may show a decrease in the user count.
  - Sometimes, events received by the MoEngage server may not be processed immediately. When such instances occur after the billing calculation, re-running the query on segmentation may show an increase in the user count.
</Accordion>

<Accordion title="What is the difference between MAUs & MTUs?">
  - **MAUs (Monthly Active Users)** represent the number of unique users who interacted with your app. MoEngage uses the _App/Site Opened_ event to calculate MAUs. This event is tracked by default as soon as a user opens an app or website.
  - **MTUs (Monthly Tracked Users)** are equivalent to each unique end-user profile that performs at least one event in a single calendar month. MoEngage counts each user only once per month, regardless of how many events they perform.
</Accordion>

<Accordion title="What metrics can I create alerts on?">
  You can create alerts for any billing-relevant metric, including MAUs, MTUs, Total Events, and all communication channels (Email, SMS, WhatsApp, etc.).
</Accordion>

<Accordion title="How soon will I know after the threshold is breached for an alert?">
  Usage monitoring runs daily. You will receive an alert notification as soon as the daily scan identifies that your usage has crossed the defined threshold.
</Accordion>

<Accordion title="Why is there a discrepancy in event count between the Billing Usage page and the Data Management page?">
  The Billing Usage page includes all data processed within the month, including historical backfills via file/data warehouse imports and late-arriving data due to any delays. In contrast, the Data Management page is a daily snapshot and does not retroactively update its historical graphs for events added after the original date has passed. Consequently, your bill will reflect the total volume of data ingested, while Data Management only shows what was captured at the time.
</Accordion>

<Accordion title="How is Inform billed? Which channels are included under Inform?">
  Inform is MoEngage's transactional messaging infrastructure. The following channels are currently available under Inform: Email, SMS, WhatsApp, and Push Notifications. Customers are billed only for channels that they use based on the number of messages sent. Sent volume for each channel is visible on the Usage page under **Settings** \> **Account** \> **Billing**. For specific pricing or volume limits associated with each channel under your Inform plan, refer to your contract or contact your CSM.
</Accordion>

<Accordion title="Why is there a discrepancy between the email volumes shown on my dashboard and the volumes I am being billed for?">
  The email volumes shown in your workspace dashboard reflect sends from that specific workspace only. However, billing for email is calculated at the **account level** across all workspaces that use the same MoEngage Sender. If the same Sender (for example, a shared "from" email address or domain or Sendgrid Subuser) is configured across multiple workspaces, the total sends from all those workspaces are aggregated for billing purposes. This means your billed email volume may be higher than what any single workspace dashboard shows. To get the full picture, use the Usage page and select all relevant workspaces.
</Accordion>

<Accordion title="Why is there a discrepancy between the SMS volumes shown on my dashboard and the volumes I am being billed for?">
  SMS billing discrepancies typically arise for two reasons:

  - **Shared shortcodes across workspaces:** Similar to email, if the same shortcode is configured across multiple workspaces, billing is aggregated at the account level. Your workspace dashboard will only show sends from that workspace, while invoicing reflects the combined total.
  - **SMS segments vs. messages:** The MoEngage dashboard shows _SMS Sent_ as the number of individual messages sent. However, billing is based on **SMS segments** — the number of individual transmission units a message is broken into. A single message that exceeds the standard GSM character limit (160 characters for plain text) is split into multiple segments, each billed separately. For example, a 320-character message counts as 2 billable segments even though it appears as 1 message in the dashboard. To reconcile the difference, review your message length distribution and check if shared shortcodes are active across multiple workspaces. Contact your CSM for a detailed segment-level billing report.
</Accordion>
