# Introduction

Push campaigns target users through notifications for your app or website. MoEngage provides the following types of push campaigns.

<Tabs>
  <Tab title="One-Time">
    One-time Push campaigns engage the user through a single message at a specific time.
  </Tab>
  <Tab title="Periodic">
    Periodic Push campaigns engage the user through a series of configured messages at a specific time and duration.
  </Tab>
  <Tab title="Event Triggered">
    Event-Triggered Push campaigns engage the user through messages sent at the right moment based on a set of events executed by a user.

    For example, if a user viewed a particular product or added it to the cart (user activity trigger) but did not make a purchase, a reminder message or a relevant offer is triggered within a few minutes of the user action to retarget.

    The most common use cases for event-triggered campaigns are:

    - Welcome messages sent to users when they install your app
    - Driving the first purchase or driving user activation
    - Abandoned cart
    - Abandoned browse
    - Subscription drop-offs
    - Payment drop-offs

    <Info>
      Event-Triggered Push campaigns can drive a maximum of 4X higher conversions than the Scheduled Push campaigns.
    </Info>
  </Tab>
  <Tab title="Device-Triggered">
    Device-Triggered Push campaigns are messages sent to the device whenever a user action triggers an event. Messages are served only to the device triggering the event and are served in real time\*\*.\*\*

    The messages are triggered only for user actions tracked on the SDK and you cannot trigger server-to-server events or events triggered at the MoEngage backend. The following are the types of Device-Triggered Push campaigns:

    - [Online user triggering an event](#h_01HKJ3FZ15JCS962EWH8YTK8SM)
    - [Offline user triggering an event](#h_01HKJ3FZ16C7R4E0PSYJKGXBT9)

    Device triggers are effectively used to bring back your users as and when they try to exit the app.

    The following delivery control methods are available while creating campaigns:

    - **Maximum times to show the message from the same campaign**: This limits the number of times a message is shown in the lifetime of a customer. This is reset when the device cache is cleared or when the app is reinstalled. This setting is not applied if set to zero, meaning that the message is shown an infinite number of times (until it fulfills other control conditions).
    - **Minimum delay between two messages from the same campaign**: To enhance the user experience, marketers can add a minimum time delay between two messages from the same campaign. If another message is generated before the delay is over, the message is not served.
    - **Expiry time for sync data**: This allows you to periodically purge the previously synced campaigns ensuring you do not end up serving old messages for long. For instance, synced campaigns are not served after the configured seven days unless there is another sync.
    - **Ignore Frequency Capping**: Used for messages that are urgent in nature and are always served to ignore the channel capping policies.
    - **Ignore DND period**: Used for messages that are urgent in nature and are always served even if the trigger is generated during the Do Not Disturb period set at the account level.
    - **Serve in offline mode**: If selected, the message is served even if a user is triggering the event when not connected to the Internet, meaning offline mode.

    ### Online Triggers

    Online triggers are served when the end user is online and executes a trigger event. All conditions are checked in real time and the user is shown the notification if eligible.

    ### Offline Triggers

    The device triggers can serve the message on a trigger event even if the end user is offline, meaning the user is not connected to the Internet.

    Offline triggers deliver the messages that are previously synced. Syncing of messages happens when the user, connected to the Internet, starts the application on which the MoEngage SDK exists.

    Offline triggers are served based on the last sync data. This ability of serving offline trigger messages has the following limitations:

    - **Segmentation**: Offline triggers attributes the user who fulfills or does not fulfill the campaign segmentation criteria during the syncs. For example, at the time of campaign sync, a user was eligible for campaign C1. Now, if the user goes offline and executes the trigger event, they are served the message.
    - **Frequency capping**: There is no cap on the number of messages served. The user is served the message as many times as the event is triggered.
    - **Personalization**: Event personalization does not work offline. Other personalization works based on the previously synced values.
    - **Campaign Performance**: Stats such as impressions and clicks for offline trigger messages are registered with the MoEngage server after the user connects to the Internet.
    - **Campaign Editing**: For an edited campaign after the initial sync, i**f the user does not open the app after the first sync and goes offline, the offline trigger is served according to the previous configuration**.

    A user's location can also help send targeted campaigns to engage them with relevant content. For instance, for a user who arrives at the airport or train station, it is important to engage the user with your hotel or taxi booking app for the city.
  </Tab>
  <Tab title=" Location Triggered">
    Location-Triggered Push campaigns are triggered for all users when they enter, exit, or stay in a pre-defined perimeter around a location (for example, the airport). Currently, Location-Triggered campaigns are available for Android and iOS platforms, and all users fulfilling the criteria of geo-fence receive the campaign.
  </Tab>
</Tabs>
