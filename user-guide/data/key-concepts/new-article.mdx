Before you build campaigns or flows, MoEngage needs to know who your users are, what they do, and how to group them. This page explains the four building blocks — user profiles, attributes, events, and segments — and how they connect.

## The four building blocks

<CardGroup cols={2}>
  <Card title="User profiles" icon="user">
    A record for each person who interacts with your app or website. Profiles are created automatically when a user opens your app or visits your site for the first time.
  </Card>
  <Card title="Attributes" icon="tag">
    Properties that describe a user — their name, location, plan type, or any custom data your team passes to MoEngage. Attributes answer the question: *who is this person?*
  </Card>
  <Card title="Events" icon="bolt">
    Actions a user takes — opening the app, viewing a product, completing a purchase. Events answer the question: *what did this person do?*
  </Card>
  <Card title="Segments" icon="users">
    Groups of users who share common attributes or behavior. Segments are dynamic — users enter and exit automatically as their data changes.
  </Card>
</CardGroup>

## How they work together

The four building blocks form a chain. Each one feeds into the next:

1. A user opens your app — MoEngage **creates a user profile**.
2. The user browses a product — MoEngage **records an event** (`Product Viewed`) and attaches event attributes like category and price.
3. Over time, the user's behavior updates their **attributes** — for example, their preferred category or last active date.
4. You define a **segment** — say, users who viewed a product in the last 7 days but haven't purchased. MoEngage evaluates all profiles against that rule and keeps the segment updated in real time.
5. You target that segment with a push notification or email campaign.

## A practical example

A retail brand wants to re-engage users who browsed shoes but didn't buy.

| Step | What happens in MoEngage |
|---|---|
| User opens the app | A user profile is created with standard attributes (device type, location, first seen date) |
| User views a product | A `Product Viewed` event is recorded with attributes: `category = shoes`, `price = 4999` |
| User exits without purchasing | No `Order Placed` event is recorded |
| Marketer creates a segment | Filter: users who did `Product Viewed` with `category = shoes` in the last 7 days AND did NOT do `Order Placed` |
| Campaign is sent | The segment resolves to 12,400 users — the campaign targets them with a discount offer |

## What MoEngage captures automatically

When you integrate the MoEngage SDK, the following data is collected without any additional configuration:

**Standard user attributes**
- First seen date, last active date, platform (iOS / Android / Web)
- Device type, operating system, app version
- Country, city (derived from IP)
- Install source, user install status

**Standard events**
- App Open / Site Visit
- App Install, App Uninstall
- Push Notification Received, Clicked, Dismissed
- Email Opened, Clicked

Everything else — product events, transaction data, custom user attributes — is configured by your development team when they integrate the SDK or via API and file imports.

<Info>
  Standard attributes and events are available in MoEngage immediately after SDK integration. You can start building segments with them before your custom event tracking is complete.
</Info>

## What you configure

Your development team sends custom data to MoEngage based on your business needs. Common examples:

- **Custom attributes** — `plan_type`, `lifetime_value`, `preferred_category`, `loyalty_tier`
- **Custom events** — `Product Viewed`, `Added to Cart`, `Order Placed`, `Loan Applied`
- **Event attributes** — properties on each event, such as `product_id`, `price`, `category`

Work with your team to define which events and attributes matter for your campaigns before integration. See [Event naming conventions](/user-guide/data/key-concepts/event-naming-conventions) for guidance.
