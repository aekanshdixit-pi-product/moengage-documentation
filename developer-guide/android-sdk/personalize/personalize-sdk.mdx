# Overview

The MoEngage Personalize SDK provides a secure framework for delivering personalized campaigns. It simplifies integration by handling user identity and authentication internally, eliminating the need to manage API secrets or manual HTTPS calls.

<Tip>
  **Prerequisites**

  Before you can fetch personalized experiences, ensure you have completed the following:

  - Ensure you have integrated the BOM into your Android SDK. For more information, refer to [Install using BOM](https://www.moengage.com/docs/developer-guide/android-sdk/sdk-integration/basic-integration/Install-Using-BOM).
  - Initialization: Ensure you have initialized the SDK within your application entry point. For more information on initialization, refer to [SDK Initialization](https://www.moengage.com/docs/developer-guide/android-sdk/sdk-integration/basic-integration/sdk-initialization).
</Tip>

# How It All Fits Together

Before writing any code, it is helpful to understand the three moving parts of the personalization workflow:

1. **Dashboard Configuration:** A marketer creates an Experience Campaign in the MoEngage dashboard and assigns it a unique `experienceKey` (e.g., `home_banner`). They configure the specific JSON payload to be returned for different user segments.
2. **The Meta Call :** Your application calls `fetchExperiencesMeta` to discover which experience keys are active and available for the current user.
3. **The Fetch Call :** Your application calls `fetchExperience` or `fetchExperiences` with a specific key to retrieve the actual payload. The SDK uses the metadata gathered in Step 2 to accurately resolve and return this request.

<Info>
    The SDK returns **raw** JSON only. As the developer, you are responsible for parsing this payload and building the corresponding UI in your application.
</Info>

# Integrate MoEngage Personalization

To begin using MoEngage Personalization in your application, once you have configured the BOM, add the required artifact in the app/build.gradle file as shown below:

```kotlin build.gradle.kts(kotlin) wrap
dependencies {
    implementation 'com.moengage:personalization-core'
}
```

# Implementation Workflow

The [`MoEPersonalizeHelper`](https://moengage.github.io/android-api-reference/personalization-core/com.moengage.campaigns.personalize/-mo-e-personalize-helper/index.html) SDK object is designed to simplify the retrieval and interaction with dynamic, personalized content. Below is a breakdown of the code divided by its primary responsibilities.

## 1. Fetch Meta Experience

Before fetching any specific payload or experience, you must invoke the metadata call. Prefetching the metadata helps you optimize the experience fetch. On the app side, you can use the metadata to identify the right personalized content for the current UI state and fetch only the relevant content instead of all the content in the application.

<Info>
  Recommended call frequency: Once per session — at app launch or immediately after user login. Do not call it before every fetchExperience call.
</Info>

<CodeGroup>

```kotlin Kotlin wrap
MoEPersonalizeHelper.fetchExperiencesMeta(
context = context,
    status = listOf(ExperienceStatus.ACTIVE),
    onSuccess = { metadata ->
        println("Success: $metadata")
    },
    onFailure = { reason, message ->
        println("Failure: $reason")
    }
)
```

```java Java wrap
MoEPersonalizeHelper.INSTANCE.fetchExperiencesMeta(
context,
    Collections.singletonList(ExperienceStatus.ACTIVE),
    metadata -> {
        System.out.println("Success: " + metadata);
    },
    (reason, message) -> {
        System.out.println("Failure: " + reason);
        return Unit.INSTANCE;
    }
);
```

</CodeGroup>

The `onSuccess` callback returns an [`ExperienceCampaignsMetadata`](https://moengage.github.io/android-api-reference/personalization-core/com.moengage.campaigns.personalize.model/-experience-campaigns-metadata/index.html) object containing the metadata required for campaign execution. Use `metadata.source` to check the `DataSource`, and access `metadata.campaignsMeta` for a list of `ExperienceCampaignMetaDetail` objects, which expose:

- `experienceKey` — the unique key of the experience campaign.
- `experienceName` — the name of the experience campaign.
- `status` — the `ExperienceStatus` of the experience campaign.

The `onFailure` callback provides an [`ExperienceFailureReason`](https://moengage.github.io/android-api-reference/personalization-core/com.moengage.campaigns.personalize.model/-experience-failure-reason/index.html) enum, which exposes:

- `entries` — a representation of an immutable list of all enum entries, in the order they're declared.
- `name` — the name of the enum entry as a `String`.
- `ordinal` — the position of the enum entry as an `Int`.

<Note>
 `fetchExperience` and `fetchExperiences` resolve keys against the result of the most recent `fetchExperiencesMeta` call. If you skip the metadata call, or call it with a status filter that does not include the keys you later request, the subsequent fetch call returns `INVALID_EXPERIENCE_KEY` in the `onFailure` callback for those keys.
</Note>

## 2. Fetch Personalized Content

Once metadata is fetched, you can retrieve the actual personalized payloads. You can fetch a single experience or multiple experiences simultaneously, with full support for contextual targeting.

<Info>
    EXPERIENCE\_KEY is the unique key that is used in the experience campaign while creating the campaign on the MoEngage dashboard. You can find this key in the `ExperienceCampaignMeta` object of the `ExperienceCampaignsMetadata` returned on successful metadata fetch.
</Info>

### Fetch Single Experience

- **Single Experience**: Retrieve a single experience using a single experience key.

<CodeGroup>

```kotlin Kotlin wrap
MoEPersonalizeHelper.fetchExperience(
    context = context,
    // EXPERIENCE_KEY must be one of the keys returned in the fetchExperiencesMeta call.
    experienceKey = EXPERIENCE_KEY,
    attributes = mapOf("customOptionalAttributeKey" to "customOptionalAttributeValue"),
    onSuccess = { campaignsResult ->
        println("Success: $campaignsResult")
    },
    onFailure = { reason, message ->
        println("Failure: $reason")
    }
)
```

```java Java wrap
MoEPersonalizeHelper.INSTANCE.fetchExperience(
    context,
	// EXPERIENCE_KEY must be one of the keys returned in the fetchExperiencesMeta call.
	EXPERIENCE_KEY,
	attributes,
	campaignsResult -> {
	    System.out.println("Success: " + campaignsResult);
	},
	(reason, message) -> {
	    System.out.println("Failure: " + reason);
	    return Unit.INSTANCE;
	}
    );
```

</CodeGroup>

### Fetch Multiple Experiences

- **Bulk Experiences**: Retrieve multiple experiences by passing an array of experience keys.

<Note>
  You can fetch up to 25 experiences in a single call. If you exceed this, the SDK returns the 25 most recently updated experiences and notifies you of the unfulfilled keys.
</Note>

<CodeGroup>

```kotlin Kotlin wrap
MoEPersonalizeHelper.fetchExperiences(
    context = context,
    // EXPERIENCE_KEY must be one of the keys returned in the fetchExperiencesMeta call.
    experienceKeys = setOf("EXPERIENCE_KEYS"),
    attributes = mapOf("customOptionalAttributeKey" to "customOptionalAttributeValue"),
    onSuccess = { campaignsResult ->
        println("Success: $campaignsResult")
    },
    onFailure = { reason, message ->
        println("Failure: $reason")
    }
)
```

```java Java wrap
Set<String> hashSet = new HashSet<>();
hashSet.add("KEY1");
hashSet.add("KEY2");

MoEPersonalizeHelper.INSTANCE.fetchExperiences(
    context,
    hashSet,
    attributes,
    campaignsResult -> {
    System.out.println("Success: " + campaignsResult);
    },
    (reason, message) -> {
	    System.out.println("Failure: " + reason);
	    return Unit.INSTANCE;
    }
);
```

</CodeGroup>

The `onSuccess` callback returns an [`ExperienceCampaignsResult`](https://moengage.github.io/android-api-reference/personalization-core/com.moengage.campaigns.personalize.model/-experience-campaigns-result/index.html) object. A single response can include both resolved and unresolved keys. Inspect the following properties:

- `experiences` — campaigns resolved successfully for the requested keys. This provides a list of `ExperienceCampaign` objects, which expose:
  - `experienceKey` — the unique key of the experience.
  - `experienceContext` — the context of the experience as a `Map<String, String>`.
  - `payload` — the payload of the experience as a `Map<String, Any>`.
  - `source` — the `DataSource` from which the experience was served.
- `failures` — a list of [`ExperienceCampaignFailure`](https://moengage.github.io/android-api-reference/personalization-core/com.moengage.campaigns.personalize.model/-experience-campaign-failure/index.html) objects representing per-key failures returned alongside the successes. Each failure object exposes:
  - `experienceKeys` — a list of the experience keys that failed.
  - `reason` — an [`ExperienceFailureReason`](https://moengage.github.io/android-api-reference/personalization-core/com.moengage.campaigns.personalize.model/-experience-failure-reason/index.html) explaining why the keys failed (e.g., SDK-side cache misses or server-side business-logic failures, such as the user being in a control group or the key having expired).

The `onFailure` callback handles system-level failures that prevent the entire request from running, such as the SDK not being initialized, network or HTTP errors, parse errors, or the edge case where every requested key is invalid (in which case the request is rejected upfront). It provides an `ExperienceFailureReason` enum detailing the exact cause of the failure.

Use Cases:

**Contextual Targeting**:  Pass an object of attributes (e.g., `{"current_page": "home", "cart_value": "500"}`) during the fetch. This enables real-time, state-dependent content delivery (e.g., showing a “Free Shipping” banner if the cart value meets a threshold).

Attribute values must be strings.

```json Example
{
  "loyalty_tier": "gold",
  "app_lang": "spanish"
}
```

<Warning>
  Do not include PII (email, phone, government IDs) in attributes. Use the dedicated user attribute APIs for identity data.
</Warning>

## 3. Notify SDK on Showing the Experience (Track Impressions)

An _impression_ is a telemetry event that notifies the MoEngage SDK that a personalized campaign payload has successfully rendered on the UI and is visible to the user.

To accurately track campaign performance, you must invoke the following methods the moment your application displays the personalized content on the screen.

### 3a. Notify SDK for Experience Campaigns

Call the `experiencesShown()` method when the UI element containing the experience renders.

<CodeGroup>

```kotlin Kotlin wrap
// campaignEntity is the campaign object returned by the fetchExperience(s) call.
// It contains the metadata and payload required by the SDK to attribute impressions and clicks to the correct campaign.

// Track a single experience:
MoEPersonalizeHelper.experienceShown(context, campaignEntity)

// Or track multiple experiences at once:
MoEPersonalizeHelper.experiencesShown(context, listOf(campaignEntity))
```

```java Java wrap
// campaignEntity is the campaign object returned by the fetchExperience(s) call.
// It contains the metadata and payload required by the SDK to attribute impressions and clicks to the correct campaign.

// Track a single experience:
MoEPersonalizeHelper.INSTANCE.experienceShown(context, campaignEntity);

// Or track multiple experiences at once:
MoEPersonalizeHelper.INSTANCE.experiencesShown(context, Arrays.asList(campaignEntity));
```

</CodeGroup>

### 3b. Notify SDK for Offering Campaigns

[Offerings](https://www.moengage.com/docs/user-guide/decisioning/offer-decisioning/offerings) are a distinct subtype of personalized content (such as a product recommendation or a coupon code). Each experience campaign can carry one or more offerings inside its `payload`. A fetched campaign that carries an offerings group has the following shape:

```json Sample ExperienceCampaign
{
  "experienceKey": "home_banner",
  "payload": {
    "<your_offering_key>": {
      "value": "[ { \"offering_content\": { ... }, \"offering_context\": { ... }, \"...\": \"...\" }, ... ]",
      "data_type": "string"
    }
  },
  "experienceContext": { ... },
  "source": "network"
}
```

The following fields appear in the sample payload:

- `payload` is the campaign's content dictionary. Offerings are nested one level below it.
- `<your_offering_key>` is configured for this campaign on the MoEngage dashboard. You define the key name when authoring the campaign.
- `value` is a **stringified JSON array**. Decode it to obtain the list of offering objects.
- Each offering object contains:
  - `offering_content` — the data you render into your UI.
  - `offering_context` — attribution data used by the SDK; do not modify it.

Once you have decoded one or more offering objects out of `campaignEntity.payload`, pass them back to the SDK when the corresponding UI renders:

<CodeGroup>

```kotlin Kotlin wrap
// offeringPayload is the data object extracted from a campaignEntity using the unique offering key assigned during campaign creation.
// It enables tracking at the individual offering level, which is essential when an experience contains multiple offerings within a single payload.

// Track a single offering:
MoEPersonalizeHelper.offeringShown(context, offeringPayload)

// Or track multiple offerings at once:
MoEPersonalizeHelper.offeringsShown(
    context,
    listOf(offeringPayload1, offeringPayload2)
)
```

```java Java wrap
// offeringPayload is the data object extracted from a campaignEntity using the unique offering key assigned during campaign creation.
// It enables tracking at the individual offering level, which is essential when an experience contains multiple offerings within a single payload.

// Track a single offering:
MoEPersonalizeHelper.INSTANCE.offeringShown(context, offeringPayload);

// Or track multiple offerings at once:
MoEPersonalizeHelper.INSTANCE.offeringsShown(
    context,
    Arrays.asList(offeringPayload1, offeringPayload2)
);
```

</CodeGroup>

## 4. Track Clicks

### 4a. Track Clicks for Experience Campaigns

Use these methods to log "Clicks" when the user interacts with the UI element.

<CodeGroup>

```kotlin Kotlin wrap
// campaignEntity is the campaign object returned by the fetchExperience(s) call.
MoEPersonalizeHelper.experienceClicked(context, campaignEntity)
```

```java Java wrap
// campaignEntity is the campaign object returned by the fetchExperience(s) call.
MoEPersonalizeHelper.INSTANCE.experienceClicked(context, campaignEntity);
```

</CodeGroup>

### 4b. Track Clicks for Offering Campaigns

Use `offeringClicked` to log a click on a specific offering item, such as a product recommendation or a coupon code, within an offering campaign. Pass both the parent `campaignEntity` and the specific offering object extracted from `campaignEntity.payload`. For the payload shape, refer to [section 3b](#3b-notify-sdk-for-offering-campaigns).

<CodeGroup>

```kotlin Kotlin wrap
// offeringPayload is the offering object the user clicked, extracted from campaignEntity.payload (decoded as shown in section 3b).
MoEPersonalizeHelper.offeringClicked(
    context,
    campaignEntity,
    offeringPayload
)
```

```java Java wrap
// offeringPayload is the offering object the user clicked, extracted from campaignEntity.payload (decoded as shown in section 3b).
MoEPersonalizeHelper.INSTANCE.offeringClicked(
    context,
    campaignEntity,
    offeringPayload
);
```

</CodeGroup>

#### Example

```JSON Sample Payload
{
  "custom_offering_key": {
    // Expected offering payload in the function call.
  }
}
```

<Info>
  - You can use these Offering-specific functions only if the campaign contains an offering payload (e.g., structured JSON data containing dynamic product recommendations, catalogs, or unique coupon codes nested under a specific offering key). For all other experience data, use the standard experience shown/clicked functions.
  - Each call sends a single offering object to the server, providing granular per-offering analytics. The object is one element from the JSON array decoded out of the `value` field inside `campaignEntity.payload`.
</Info>

# FAQs

<Accordion title="Does the SDK handle UI rendering?">
  No. The SDK returns raw JSON payloads. You are responsible for parsing the data and building the UI components (e.g., banners or carousels).
</Accordion>

<Accordion title="Does the SDK cache images or videos?">
  No. The SDK does not download or cache media assets. Use a standard media loading library to handle images, videos, or fonts referenced in the JSON.
</Accordion>

<Accordion title="Is there a limit on how many experiences I can fetch?">
  Yes. You can fetch up to 25 experiences in a single call. If you exceed this, the SDK returns the 25 most recently updated experiences and notifies you of the unfulfilled keys.
</Accordion>

<Accordion title="What happens if a fetch request is made while the device is offline and no valid cache is available? ">
  The SDK returns an empty payload along with a standardized error code (e.g., NETWORK\_ERROR). For more information on all the errors, refer [here](https://moengage.github.io/android-api-reference/personalization-core/com.moengage.campaigns.personalize.model/-experience-failure-reason/index.html).
</Accordion>
