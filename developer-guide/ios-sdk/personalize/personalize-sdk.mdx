# Overview

The MoEngage iOS SDK provides a secure framework for delivering personalized campaigns. It simplifies integration by handling user identity and authentication internally, eliminating the need to manage API secrets or manual HTTPS calls. This allows you to fetch and track personalized campaign information directly through a streamlined, native interface.

<Tip>
  Prerequisite

  Before you can fetch personalized experiences, ensure you have called the SDK `initialize()` method within your application entry point ( `didFinishLaunchingWithOptions` on iOS). For more information, refer to [SDK initialization](https://www.moengage.com/docs/developer-guide/ios-sdk/sdk-integration/basic/sdk-initialization).
</Tip>

# Implementation Workflow

The `MoEngageSDKPersonalize` singleton is designed to simplify the retrieval and interaction with dynamic, personalized content on iOS. The SDK provides native Swift methods as well as Objective-C-compatible overloads.

## 1. Fetching Meta Experience

{/* SDK-TEAM-SUGGESTION:
Section header style — every other section in this doc uses imperative
("Fetch Personalized Content", "Track Impressions"). Please rename this
section to "Fetch Meta Experience" for consistency (also matches the Android doc).
*/}

Before fetching any specific payload or experience, you must invoke the metadata call. This step provides the SDK with the necessary configuration and context to accurately process subsequent requests. The SDK offers dedicated methods for both Swift and Objective-C.

<CodeGroup>

```swift Swift wrap
import MoEngagePersonalization

// All fetch and tracking methods accept an optional trailing `workspaceId`.
// Multi-workspace apps pass the workspace (App ID); single-workspace apps
// can omit it (defaults to nil → the default workspace).
MoEngageSDKPersonalize.sharedInstance.fetchExperiencesMeta(
    status: [.active],
    onSuccess: { metadata in
        // metadata.source                  — .cache or .network
        // metadata.experienceCampaignMeta — [MoEngageExperienceCampaignMetaDetail]
        print("Fetched \(metadata.experienceCampaignMeta.count) experience(s)")
    },
    onFailure: { reason in
        print("Meta fetch failed: \(reason.displayCode) — \(reason.message)")
    },
    workspaceId: nil
)
```

```objective-c Objective-C wrap
@import MoEngagePersonalization;

NSArray<NSNumber *> *statuses = @[ @(MoEngageExperienceStatusActive) ];

[[MoEngageSDKPersonalize sharedInstance]
    fetchExperiencesMetaWithStatusRawValues:statuses
    onSuccess:^(MoEngageExperienceCampaignMetaData *metadata) {
        NSLog(@"Fetched %lu experience(s)",
              (unsigned long)metadata.experienceCampaignMeta.count);
    }
    onFailure:^(MoEngageExperienceFailureReason *reason) {
        NSLog(@"Meta fetch failed: %@ — %@", reason.displayCode, reason.message);
    }
    workspaceId:nil];
```

</CodeGroup>

The `onSuccess` callback returns an [`ExperienceCampaignsMetadata`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@objc(cs)MoEngageExperienceCampaignMetaData) object containing all necessary metadata for campaign execution. Conversely, the `onFailure` callback provides a [`RequestFailureReasonCode`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@E@MoEngageExperienceFailureReasonCode) and an optional message to identify the specific reason for the request's failure.

{/* SDK-TEAM-SUGGESTION:

Always call the metadata fetch first to initialize core configurations based on the desired experience status. Use the standard method for Swift arrays, or the ObjC variant if passing raw NSNumber values from Objective-C code.

## 2. Fetch Personalized Content

Once metadata is fetched, you can retrieve the actual personalized payloads. You can fetch a single experience or multiple experiences simultaneously, with full support for contextual targeting.

{/* SDK-TEAM-SUGGESTION:
Consider splitting this section into "### Fetch a Single Experience" and

  <Info>
    `experienceKey` is the unique key configured for the campaign on the
    MoEngage dashboard. You can discover the available keys in the

    made with a key not present in the meta result fail with
    `.invalidExperienceKey`.
  </Info>

The code snippet below to fetch a single experience is compatible with both Swift and Objective-C implementations:

{/* SDK-TEAM-SUGGESTION:
The line above reads as if a single snippet works for both Swift and ObjC, but
the block below actually has separate Swift and Objective-C tabs. Suggested
replacement:
  "The Swift and Objective-C snippets below fetch a single experience."
*/}

<CodeGroup>

```swift Swift wrap
let attributes: [String: String] = [
    "current_page": "home",
    "cart_value": "500"
]

// `attributes` is optional — use it for contextual targeting; omit the parameter entirely to skip.
MoEngageSDKPersonalize.sharedInstance.fetchExperience(
    experienceKey: "home_banner",
    attributes: attributes,
    onSuccess: { result in
        // result.experiences — campaigns resolved for this key
        // result.failures    — per-key business-logic failures
        print("Single experience: \(result)")
    },
    onFailure: { reason in
        print("Fetch failed: \(reason.displayCode) — \(reason.message)")
    }
)
```

```objective-c Objective-C wrap
NSDictionary<NSString *, NSString *> *attributes = @{
    @"current_page": @"home",
    @"cart_value": @"500"
};

// `attributes` is optional — use it for contextual targeting; pass @{} to skip.
[[MoEngageSDKPersonalize sharedInstance]
    fetchExperienceWithExperienceKey:@"home_banner"
    attributes:attributes
    onSuccess:^(MoEngageExperienceCampaignsResult *result) {
        NSLog(@"Single experience: %@", result);
    }
    onFailure:^(MoEngageExperienceFailureReason *reason) {
        NSLog(@"Fetch failed: %@", reason.message);
    }
    workspaceId:nil];
```

</CodeGroup>

To fetch multiple experiences:

<CodeGroup>

```swift Swift wrap
// EXPERIENCE_KEY must be one of the keys returned in the fetchExperiencesMeta call.
// Using a key that is not part of the meta call will result in an invalid key error.
// `attributes` is optional — use it for contextual targeting; omit the parameter entirely to skip.
MoEngageSDKPersonalize.sharedInstance.fetchExperiences(
    experienceKeys: ["home_banner", "checkout_offer"],
    attributes: attributes,
    onSuccess: { result in
        // result.experiences — successfully resolved campaigns
        // result.failures    — per-key failures (e.g. user not in segment for one key)
    },
    onFailure: { reason in
        print("Fetch failed: \(reason.displayCode) — \(reason.message)")
    }
)
```

```objective-c Objective-C wrap
// EXPERIENCE_KEY must be one of the keys returned in the fetchExperiencesMeta call.
// Using a key that is not part of the meta call will result in an invalid key error.
// `attributes` is optional — use it for contextual targeting; pass @{} to skip.
[[MoEngageSDKPersonalize sharedInstance]
    fetchExperiencesWithKeys:@[@"home_banner", @"checkout_offer"]
    attributes:attributes
    onSuccess:^(MoEngageExperienceCampaignsResult *result) {
        NSLog(@"%lu succeeded, %lu failed",
              (unsigned long)result.experiences.count,
              (unsigned long)result.failures.count);
    }
    onFailure:^(MoEngageExperienceFailureReason *reason) {
        NSLog(@"Fetch failed: %@", reason.message);
    }
    workspaceId:nil];
```

</CodeGroup>

The `onSuccess` callback returns an  [`ExperienceCampaignsResult`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@objc(cs)MoEngageExperienceCampaignsResult) object containing all necessary metadata for campaign execution. Conversely, the `onFailure` callback provides a [`RequestFailureReasonCode`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@E@MoEngageExperienceFailureReasonCode) and an optional message to identify the specific reason for the request's failure.

{/* SDK-TEAM-SUGGESTION:
Consider being explicit about the split returned by
ExperienceCampaignsResult — customers often miss that some keys may succeed

  - `experiences` — campaigns resolved successfully.
  - `failures`    — per-key failures, including BOTH:

  - "RequestFailureReasonCode" → `MoEngageExperienceFailureReason` (for the

You can fetch :

- Single Experiences: Retrieve a specific payload using a single experienceKey.
- Bulk Experiences: Retrieve multiple payloads at once. In Swift, pass a Set\<String\>. For Objective-C, use the `fetchExperiencesObjC` method which accepts an NSArray bridged as \[String\].

{/* SDK-TEAM-SUGGESTION:
The "fetchExperiencesObjC" name in the bullet above is the Swift FUNCTION

exposed (via @objc(...) on the Swift source) is
`fetchExperiencesWithKeys:attributes:onSuccess:onFailure:workspaceId:`, which
takes `NSArray<NSString *>`. Suggested rewrite of the bullet:

  - Bulk Experiences: Retrieve multiple payloads at once. In Swift, pass a
    Set\<String\>. In Objective-C, call
    `fetchExperiencesWithKeys:attributes:onSuccess:onFailure:workspaceId:`,
    which accepts an `NSArray<NSString *>`.

Use cases:

**Contextual Targeting**: Pass a dictionary of attributes (e.g., \["current\_page": "home", "cart\_value": "500"\]) during the fetch. This enables real-time, state-dependent content delivery (e.g., showing a "Free Shipping" banner if the cart value meets a threshold).

## 3. Track Impressions

To accurately measure campaign performance, you must track user interactions after rendering the personalized content on the UI. The below-mentioned tracking code snippets are compatible with both Swift and Objective-C implementations.

### 3a. Track Impressions for Experience Campaigns

<CodeGroup>

```swift Swift wrap
// `campaign` is one of the objects returned in result.experiences from a fetch call.
// Pass the exact same instance back to the tracking call.
MoEngageSDKPersonalize.sharedInstance.experienceShown(campaign: campaign)

// Or track several campaigns at once:
MoEngageSDKPersonalize.sharedInstance.experiencesShown(campaigns: campaigns)
```

```objective-c Objective-C wrap
// `campaign` / `campaigns` are objects returned from a fetch call.
[[MoEngageSDKPersonalize sharedInstance] experienceShownWithCampaign:campaign workspaceId:nil];

[[MoEngageSDKPersonalize sharedInstance] experiencesShownWithCampaigns:campaigns workspaceId:nil];
```

</CodeGroup>

You can use these methods to log "Impressions" (`experienceShown / experiencesShown`) when the UI renders the content.

### 3b. Track Impressions for Offering Campaigns

[Offerings](https://www.moengage.com/docs/user-guide/decisioning/offer-decisioning/offerings) are a distinct subtype of personalized content (such as a product recommendation or a coupon code). Each experience campaign can carry one or more offerings inside its `payload`. A fetched campaign that carries an offerings group has the following shape:

```json Sample MoEngageExperienceCampaign
{
  "experienceKey": "home_banner",
  "payload": {
    "<your_offering_key>": {
      "value": "[ { \"offering_content\": { ... }, \"offering_context\": { ... }, \"...\": \"...\" }, ... ]",
      "data_type": "string"
    }
  },
  "experienceContext": { ... },
  "source": "network"
}
```

- `payload` is the campaign's content dictionary — offerings live one level below it.
- `<your_offering_key>` is configured for this campaign on the MoEngage dashboard — you decide its name when authoring the campaign.
- `value` is a **stringified JSON array**. Decode it to obtain the list of offering dicts.
- Each offering dict contains:
  - `offering_content` — the data you render into your UI.
  - `offering_context` — attribution data used by the SDK; do not modify it.

Once you have decoded one or more offering dicts out of `campaign.payload`, pass them back to the SDK when the corresponding UI renders:

<CodeGroup>

```swift Swift wrap
// `offeringPayload` is one decoded offering dict from campaign.payload.
// `offeringPayloads` is an array of such dicts.
let offeringPayload: [String: Any] = [:]
let offeringPayloads: [[String: Any]] = []

// Track a single offering:
MoEngageSDKPersonalize.sharedInstance.offeringShown(offeringPayload: offeringPayload)

// Or track multiple offerings at once:
MoEngageSDKPersonalize.sharedInstance.offeringsShown(offeringPayloads: offeringPayloads)
```

```objective-c Objective-C wrap
// `offeringPayload` is one decoded offering dict from campaign.payload.
// `offeringPayloads` is an array of such dicts.
[[MoEngageSDKPersonalize sharedInstance]
    offeringShownWithOfferingPayload:offeringPayload
    workspaceId:nil];

[[MoEngageSDKPersonalize sharedInstance]
    offeringsShownWithOfferingPayloads:offeringPayloads
    workspaceId:nil];
```

</CodeGroup>

# FAQs

<Accordion title="Does the SDK handle UI rendering?">
  No. The SDK returns raw JSON payloads. You are responsible for parsing the data and building the UI components (e.g., banners or carousels).
</Accordion>

<Accordion title="Does the SDK cache images or videos?">
  No. The SDK does not download or cache media assets. Use a standard media loading library to handle images, videos, or fonts referenced in the JSON.
</Accordion>

<Accordion title="Is there a limit on how many experiences I can fetch?">
  Yes. You can fetch up to 25 experiences in a single call. If you exceed this, the SDK returns the 25 most recently updated experiences and notifies you of the unfulfilled keys.
</Accordion>

<Accordion title="What happens if a fetch request is made while the device is offline and no valid cache is available? ">
  The SDK returns an empty payload along with a standardized error code (e.g., NETWORK\_ERROR). For more information on all the errors, refer [here](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@objc(cs)MoEngageExperienceFailureReason).
</Accordion>
