# Overview

The MoEngage Personalize SDK provides a secure framework for delivering personalized campaigns. It simplifies integration by handling user identity and authentication internally, eliminating the need to manage API secrets or manual HTTPS calls.

<Tip>
  **Prerequisite**

  Before fetching personalized experiences, ensure the core MoEngage SDK is initialized in your application, typically inside `application(_:didFinishLaunchingWithOptions:)`. Use `MoEngage.sharedInstance.initializeDefaultLiveInstance(_:)` for production or `initializeDefaultTestInstance(_:)` for development. For more information, refer to [SDK initialization](https://www.moengage.com/docs/developer-guide/ios-sdk/sdk-integration/basic/sdk-initialization).
</Tip>

# How It All Fits Together

Before writing any code, it is helpful to understand the three moving parts of the personalization workflow:

1. **Dashboard Configuration:** A marketer creates an Experience Campaign in the MoEngage dashboard and assigns it a unique `experienceKey` (e.g., `home_banner`). They configure the specific JSON payload to be returned for different user segments.
2. **The Meta Call :** Your application calls `fetchExperiencesMeta` to discover which experience keys are active and available for the current user.
3. **The Fetch Call :** Your application calls `fetchExperience` or `fetchExperiences` with a specific key to retrieve the actual payload. The SDK uses the metadata gathered in Step 2 to accurately resolve and return this request.

<Info>
  Â Â The SDK returns **raw** JSON only. As the developer, you are responsible for parsing this payload and building the corresponding UI in your application.
</Info>

# Implementation Workflow

The `MoEngageSDKPersonalize` singleton is designed to simplify the retrieval and interaction with dynamic, personalized content on iOS. The SDK provides native Swift methods as well as Objective-C-compatible overloads.

<Info>
  All fetch and tracking methods accept an optional `workspaceId` as the final argument. Multi-workspace apps pass the workspace App ID with each call. Single-workspace apps can omit it; passing `nil` resolves to the default workspace.
</Info>

## 1. Fetch Meta Experience

Before fetching any specific payload or experience, you must invoke the metadata call. Prefetching the metadata helps you optimize the experience fetch. On the app side, you can use the metadata to identify the right personalized content for the current UI state and fetch only the relevant content instead of all the content in the application.

<CodeGroup>

```swift Swift wrap
// All fetch and tracking methods accept an optional trailing `workspaceId`.
// Multi-workspace apps pass the workspace (App ID); single-workspace apps
// can omit it (defaults to nil â†’ the default workspace).
MoEngageSDKPersonalize.sharedInstance.fetchExperiencesMeta(
    status: [.active],
    onSuccess: { metadata in
        // metadata.source                  â€” .cache or .network
        // metadata.experienceCampaignMeta â€” [MoEngageExperienceCampaignMetaDetail]
        print("Fetched \(metadata.experienceCampaignMeta.count) experience(s)")
    },
    onFailure: { reason in
        print("Meta fetch failed: \(reason.displayCode) â€” \(reason.message)")
    },
    workspaceId: nil
)
```

```objective-c Objective-c wrap
@import MoEngagePersonalization;

NSArray<NSNumber *> *statuses = @[ @(MoEngageExperienceStatusActive) ];

[[MoEngageSDKPersonalize sharedInstance]
    fetchExperiencesMetaWithStatusRawValues:statuses
    onSuccess:^(MoEngageExperienceCampaignMetaData *metadata) {
        NSLog(@"Fetched %lu experience(s)",
              (unsigned long)metadata.experienceCampaignMeta.count);
    }
    onFailure:^(MoEngageExperienceFailureReason *reason) {
        NSLog(@"Meta fetch failed: %@ â€” %@", reason.displayCode, reason.message);
    }
    workspaceId:nil];
```

</CodeGroup>

The `onSuccess` callback returns a [`MoEngageExperienceCampaignMetaData`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@objc(cs)MoEngageExperienceCampaignMetaData) object containing the metadata required for campaign execution. Use `metadata.source` to check whether the data was served from `.cache` or fetched from `.network`.

The `onFailure` callback provides a [`MoEngageExperienceFailureReason`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@objc(cs)MoEngageExperienceFailureReason), which exposes:

- `displayCode` â€” a human-readable error code such as `HTTP404` or `E006`.
- `httpStatusCode` â€” the underlying HTTP status code, when the failure originates from a network request.
- `message` â€” a description of the failure.

The code property on `MoEngageExperienceFailureReason` is a [`MoEngageExperienceFailureReasonCode`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@E@MoEngageExperienceFailureReasonCode) enum value.

<Note>
  Pass `status: []` to return experiences across all statuses.
</Note>

## 2. Fetch Personalized Content

Once metadata is fetched, you can retrieve the actual personalized payloads. You can fetch a single experience or multiple experiences simultaneously, with full support for contextual targeting.

<Info>
  - EXPERIENCE\_KEY is the unique key that is used in the experience campaign while creating the campaign on the MoEngage dashboard. You can find this key in the `ExperienceCampaignMeta` object of the `ExperienceCampaignsMetadata` returned on successful metadata fetch.
  - A fetch call made with a key that is not present in the metadata result fails with `.invalidExperienceKey`.
</Info>

### Fetch Single Experience

- Single Experiences: Retrieve a single experience using a single experience key.

<CodeGroup>

```swift Swift wrap
let attributes: [String: String] = [
    "current_page": "home",
    "cart_value": "500"
]

// `attributes` is optional â€” use it for contextual targeting; omit the parameter entirely to skip.
MoEngageSDKPersonalize.sharedInstance.fetchExperience(
    experienceKey: "home_banner",
    attributes: attributes,
    onSuccess: { result in
        // result.experiences â€” campaigns resolved for this key
        // result.failures    â€” per-key business-logic failures
        print("Single experience: \(result)")
    },
    onFailure: { reason in
        print("Fetch failed: \(reason.displayCode) â€” \(reason.message)")
    }
)
```

```objective-c Objective-c wrap
NSDictionary<NSString *, NSString *> *attributes = @{
    @"current_page": @"home",
    @"cart_value": @"500"
};

// `attributes` is optional â€” use it for contextual targeting; pass @{} to skip.
[[MoEngageSDKPersonalize sharedInstance]
    fetchExperienceWithExperienceKey:@"home_banner"
    attributes:attributes
    onSuccess:^(MoEngageExperienceCampaignsResult *result) {
        NSLog(@"Single experience: %@", result);
    }
    onFailure:^(MoEngageExperienceFailureReason *reason) {
        NSLog(@"Fetch failed: %@", reason.message);
    }
    workspaceId:nil];
```

</CodeGroup>

### Fetch Multiple Experiences

- Bulk Experiences: Retrieve multiple experiences by passing a collection of experience keys. In Swift, pass a `Set<String>`. In Objective-C, call `fetchExperiencesWithKeys:attributes:onSuccess:onFailure:workspaceId:`, which accepts an `NSArray<NSString *>`.

<Note>
  You can fetch up to 25 experiences in a single call. If you exceed this, the SDK returns the 25 most recently updated experiences and notifies you of the unfulfilled keys.
</Note>

<CodeGroup>

```swift Swift wrap
// EXPERIENCE_KEY must be one of the keys returned in the fetchExperiencesMeta call.
// Using a key that is not part of the meta call will result in an invalid key error.
// `attributes` is optional â€” use it for contextual targeting; omit the parameter entirely to skip.
MoEngageSDKPersonalize.sharedInstance.fetchExperiences(
    experienceKeys: ["home_banner", "checkout_offer"],
    attributes: attributes,
    onSuccess: { result in
        // result.experiences â€” successfully resolved campaigns
        // result.failures    â€” per-key failures (e.g. user not in segment for one key)
    },
    onFailure: { reason in
        print("Fetch failed: \(reason.displayCode) â€” \(reason.message)")
    }
)
```

```objective-c Objective-c wrap
// EXPERIENCE_KEY must be one of the keys returned in the fetchExperiencesMeta call.
// Using a key that is not part of the meta call will result in an invalid key error.
// `attributes` is optional â€” use it for contextual targeting; pass @{} to skip.
[[MoEngageSDKPersonalize sharedInstance]
    fetchExperiencesWithKeys:@[@"home_banner", @"checkout_offer"]
    attributes:attributes
    onSuccess:^(MoEngageExperienceCampaignsResult *result) {
        NSLog(@"%lu succeeded, %lu failed",
              (unsigned long)result.experiences.count,
              (unsigned long)result.failures.count);
    }
    onFailure:^(MoEngageExperienceFailureReason *reason) {
        NSLog(@"Fetch failed: %@", reason.message);
    }
    workspaceId:nil];
```

</CodeGroup>

The `onSuccess` callback returns a [`MoEngageExperienceCampaignsResult`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@objc(cs)MoEngageExperienceCampaignsResult) object. A single response can include both resolved and unresolved keys. Inspect the following properties:

- `experiences` â€” campaigns resolved successfully for the requested keys.
- `failures` â€” per-key failures returned alongside the successes. This includes:
  - SDK-side `.invalidExperienceKey` entries for any requested keys not present in the metadata cache.
  - Server-side business-logic failures, such as the user being in a control group, not in the campaign segment, or the key having expired.

The `onFailure` callback handles system-level failures that prevent the entire request from running, such as the SDK not being initialized, network or HTTP errors, parse errors, or the edge case where every requested key is invalid (in which case the request is rejected upfront). It provides a [`MoEngageExperienceFailureReason`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@objc(cs)MoEngageExperienceFailureReason), whose `.code` property is a [`MoEngageExperienceFailureReasonCode`](https://moengage.github.io/ios-api-reference/MoEngagePersonalization.html#/c:@M@MoEngagePersonalization@E@MoEngageExperienceFailureReasonCode) enum value.

Use cases:

**Contextual Targeting**: Pass a dictionary of attributes (e.g., \["current\_page": "home", "cart\_value": "500"\]) during the fetch. This enables real-time, state-dependent content delivery (e.g., showing a "Free Shipping" banner if the cart value meets a threshold).

Attribute values must be strings.

```json Example
{
  "loyalty_tier": "gold",
  "app_lang": "spanish"
}
```

<Warning>
  Do not include PII (email, phone, government IDs) in attributes. Use the dedicated user attribute APIs for identity data.
</Warning>

## 3. Notify SDK on Showing the Experience (Track Impressions)

An _impression_ is a telemetry event that notifies the MoEngage SDK that a personalized campaign payload has successfully rendered on the UI and is visible to the user.

To accurately track campaign performance, you must invoke the following methods the moment your application displays the personalized content on the screen.

### 3a. Notify SDK for Experience Campaigns

Call the `experiencesShown()` method when the UI element containing the experience renders.

<CodeGroup>

```swift Swift wrap
// `campaign` is one of the objects returned in result.experiences from a fetch call.
// Pass the exact same instance back to the tracking call.
MoEngageSDKPersonalize.sharedInstance.experienceShown(campaign: campaign)

// Or track several campaigns at once:
MoEngageSDKPersonalize.sharedInstance.experiencesShown(campaigns: campaigns)
```

```objective-c Objective-c wrap
// `campaign` is a single object returned in result.experiences from a fetch call.
// `campaigns` is an array of such objects. Pass the exact same instance(s) back to the tracking call.

// Track a single experience:
[[MoEngageSDKPersonalize sharedInstance] experienceShownWithCampaign:campaign workspaceId:nil];

// Or track multiple experiences at once:
[[MoEngageSDKPersonalize sharedInstance] experiencesShownWithCampaigns:campaigns workspaceId:nil];
```

</CodeGroup>

### 3b. Notify SDK for Offering Campaigns

[Offerings](https://www.moengage.com/docs/user-guide/decisioning/offer-decisioning/offerings) are a distinct subtype of personalized content (such as a product recommendation or a coupon code). Each experience campaign can carry one or more offerings inside its `payload`. A fetched campaign that carries an offerings group has the following shape:

```json Sample MoEngageExperienceCampaign
{
  "experienceKey": "home_banner",
  "payload": {
    "<your_offering_key>": {
      "value": "[ { \"offering_content\": { ... }, \"offering_context\": { ... }, \"...\": \"...\" }, ... ]",
      "data_type": "string"
    }
  },
  "experienceContext": { ... },
  "source": "network"
}
```

The following fields appear in the sample payload:

- `payload` is the campaign's content dictionary. Offerings are nested one level below it.
- `<your_offering_key>` is configured for this campaign on the MoEngage dashboard. You define the key name when authoring the campaign.
- `value` is a **stringified JSON array**. Decode it to obtain the list of offering dictionaries.
- Each offering dictionary contains:
  - `offering_content` â€” the data you render into your UI.
  - `offering_context` â€” attribution data used by the SDK; do not modify it.

Once you have decoded one or more offering dictionaries out of `campaign.payload`, pass them back to the SDK when the corresponding UI renders:

<CodeGroup>

```swift Swift wrap
// `offeringPayload` is one decoded offering dict from campaign.payload.
// `offeringPayloads` is an array of such dicts.
let offeringPayload: [String: Any] = [:]
let offeringPayloads: [[String: Any]] = []

// Track a single offering:
MoEngageSDKPersonalize.sharedInstance.offeringShown(offeringPayload: offeringPayload)

// Or track multiple offerings at once:
MoEngageSDKPersonalize.sharedInstance.offeringsShown(offeringPayloads: offeringPayloads)
```

```objective-c Objective-c wrap
// `offeringPayload` is one decoded offering dict from campaign.payload.
// `offeringPayloads` is an array of such dicts.
[[MoEngageSDKPersonalize sharedInstance]
    offeringShownWithOfferingPayload:offeringPayload
    workspaceId:nil];

[[MoEngageSDKPersonalize sharedInstance]
    offeringsShownWithOfferingPayloads:offeringPayloads
    workspaceId:nil];
```

</CodeGroup>

## 4. Track Clicks

### 4a. Track Clicks for Experience Campaigns

Use these methods to log "Clicks" when the user interacts with the UI element.

<CodeGroup>

```swift Swift wrap

```

```objective-c Objective-c wrap
[[MoEngageSDKPersonalize sharedInstance] experienceClickedWithCampaign:campaign workspaceId:nil];
```

</CodeGroup>

### 4b. Track Clicks for Offering Campaigns

Use `offeringClicked` to log a click on a specific offering item, such as a product recommendation or a coupon code, within an offering campaign. Pass both the parent `campaign` and the specific offering dictionary extracted from `campaign.payload`. For the payload shape, refer to [section 3b](#3b-notify-sdk-for-offering-campaigns).


<CodeGroup>

```swift Swift wrap

// (decoded from campaign.payload as shown in Â§3b).

    campaign: campaign,
    offeringPayload: offeringPayload
)
```

```objective-c Objective-c wrap
// `offeringPayload` is the decoded offering dict for the offering the user just clicked
// (decoded from campaign.payload as shown in Â§3b).

    offeringClickedWithCampaign:campaign

    workspaceId:nil];
```

# FAQs

<Accordion title="Does the SDK handle UI rendering?">
  No. The SDK returns raw JSON payloads. You are responsible for parsing the data and building the UI components (e.g., banners or carousels).
</Accordion>

<Accordion title="Does the SDK cache images or videos?">
  No. The SDK does not download or cache media assets. Use a standard media loading library to handle images, videos, or fonts referenced in the JSON.
</Accordion>

</Accordion>
