# Overview

The _Web Assist Plugin_ is a diagnostic tool that integrates directly into your website to report SDK-related errors, data, user tracking information, and campaign details. Unlike previous methods that only displayed data post-integration or required raising Support Engineering (SE) tickets, this plugin proactively validates integration points to significantly reduce delays in go-live.

<Info> 

**Key Capabilities**

The plugin actively monitors and proactively reports on the following:

- **Service Worker & Web Push:** Detects registration failures or missing files.
- **SDK Error Tracking:** Captures handled and unhandled SDK errors. As part of the native SDK capabilities, the Logger module actively emits events for selected warnings and errors, which are then captured by the plugin.
- **Browser Storage Alerts:** Listens for browser storage clear alerts. Specifically, the plugin monitors crucial SDK storage keys such as `[INSERT_STORAGE_KEY_1]` and `[INSERT_STORAGE_KEY_2]`.
- **Lifecycle Events:** Tracks SDK events via standard MoEngage lifecycle hooks.

</Info>

# Architecture and DOM Injection

This section covers the architectural setup and how the plugin renders within your project.

The Web Assist Plugin operates directly within the parent website's environment:

1. **Extension Detection**: The Web SDK is detected on the website using the Chrome extension logic (`moeSdkDetector.js`).
2. **Script Injection**: Once detected, the Web Assist script is injected directly into the DOM.
3. **Native Element Rendering**: The Web Assist UI forms a native `<div>` on the client's website, and not an `<iframe>`. This specific approach allows the plugin to directly access the parent website’s storage and other properties.
4. **Existing Logic**: Web Assist utilizes the existing Assist extension logic for the Setup Info, Track, and Campaign sections.

# Implementing the Web Assist Plugin

Once injected, the plugin automatically adds several listeners and validation checks to monitor your environment.

## Step 1: Understand Event Listeners

Upon initialization, the plugin adds standard listeners to the `window` to monitor the environment for errors, unhandled rejections, lifecycle events, and storage modifications. This works in tandem with the Logger module emitting warnings and errors.

<CodeGroup>

```javascript JavaScript wrap
window.onerror = function(message, source, lineno, colno, error) {
  issueHandler('error', {message, source, lineno, colno, error})
};

window.addEventListener('unhandledrejection', function(event) {
  issueHandler('unhandledrejection', {event})
});

window.addEventListener('MOE_LIFECYCLE', function(event) {
  issueHandler('moeLifecycle', {event})
});

window.addEventListener('storage', function(event) {
  // If the key is null, it means the item was deleted or the storage was cleared
  // This is where keys like [INSERT_STORAGE_KEY_1] are monitored
});
```

</CodeGroup>

## Step 2: Service Worker Validation

To detect service worker issues, the plugin checks if the service worker is active and correctly placed at the root directory.

<CodeGroup>

```javascript JavaScript wrap
async function checkServiceWorker(swPath) {
  try {
      const registrations = await navigator.serviceWorker.getRegistrations();
      const moeSW = registrations.find(r => r.active && r.active.scriptURL.includes(swPath));
      
      if (!moeSW) {
          reportIssue({
              type: 'Setup',
              title: 'Service worker file missing',
              desc: 'Service worker register failed. Check if sw.js is at the root.',
              link: 'Navigate to Setup info',
              timestamp: new Date().getTime(),
          });
      }
  } catch (err) {
      // Browser might not support Service Workers or blocked by security
  }
}
```

</CodeGroup>

## Step 3: Triggering the Assist UI

You can trigger the Web Assist UI programmatically using the public SDK method.

<Note> 

**Extension Check**

If the Chrome extension is already installed, calling this method will do nothing and simply log a warning message advising the developer to use the extension instead.

</Note>

<CodeGroup>

```javascript JavaScript wrap
Moengage.enableAssist();
```

</CodeGroup>

# Testing and Validation

Once your app is configured, you can run tests in your development or staging environment to ensure the Web Assist Plugin is successfully catching integration issues.

### Test Standard Errors

Open your browser console and manually throw an error to trigger the `window.onerror` listener. The Assist UI should capture and report this as a handled/unhandled error.

<CodeGroup>

```javascript JavaScript wrap
throw new Error("Testing Web Assist Error Listener");
```

</CodeGroup>

### Test Promise Rejections

Trigger an unhandled promise rejection in the console to test the `unhandledrejection` listener. The Assist UI should log this rejection.

<CodeGroup>

```javascript JavaScript wrap
Promise.reject(new Error("Testing Promise Rejection"));
```

</CodeGroup>

### Test Storage Alerts

Open your browser's Developer Tools, navigate to the Application/Storage tab, and manually delete `[INSERT_STORAGE_KEY_1]` or `[INSERT_STORAGE_KEY_2]` from Local Storage.

<Check> 

**Success/Developer Info**

The `storage` event listener should detect the key becoming null and successfully trigger a browser storage clear alert in the plugin. 

</Check>

### Test Service Worker Validation

Temporarily rename your `sw.js` file (or move it out of the root directory) and reload the page. The `checkServiceWorker` function should fail to find the active script URL matching the path, and the plugin should report a "Setup" issue stating: "Service worker register failed. Check if sw.js is at the root."
